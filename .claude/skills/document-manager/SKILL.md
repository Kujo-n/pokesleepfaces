# ドキュメント作成・管理

ポケモンスリープ寝顔図鑑管理アプリ（Next.js + Firebase）の `docs/` を保守する作業を担うスキルです。目指すのは、**技術的に正確で、既存ドキュメントと一貫性があり、コードの実態とズレないドキュメント**。読み手は将来このリポジトリを触る開発者（多くは数か月後の自分自身）です。彼らが迷わず作業を再開できることを最終的な品質基準にしてください。

## 最優先の原則

- **言語は日本語**。すべての出力・ドキュメント本文を日本語で書きます。コード識別子・コマンド・ファイルパスはそのまま英数字で構いません。
- **書く前に読む**。既存ドキュメントを確認し、用語・章立て・トーンを揃えてから書きます。新しい流儀を持ち込むより、既存の流儀に乗るほうが読み手に優しいからです。
- **実態を正とする**。命名規則や「必須ドキュメント一覧」を記憶や思い込みで決めず、その時点の `docs/` を実際に見て判断します。プロジェクトは育つので、固定リストは陳腐化します。

## 作業フロー

### 1. 既存資料を参照して文脈を掴む

新規・更新どちらの作業でも、まず関連する既存ドキュメントに目を通します。

- **フォルダ構成・ファイル配置**: `docs/1_folder-structure.md` を参照し、それに従います。新しいファイルやディレクトリを追加・移動した場合は、このドキュメントの記載と矛盾していないか確認します。
- **アーキテクチャ・システム構成**: `docs/3_architecture.md` を参照し、図や構成説明と矛盾しないようにします。
- **用語・フォーマットの統一**: 上記以外でも、書こうとしている主題に近い既存ドキュメント（例: パフォーマンスの話なら `docs/4_performance.md`）を読み、同じ用語・同じ見出しの粒度で書きます。

`docs/` の現在のラインアップは固定ではありません。作業のたびに `docs/` を一覧し、連番の最大値・命名の付け方・索引の構造を実際に確認してください。

### 2. 命名と配置

`docs/` 配下のファイルは **`数字_説明.md`** の規則に従います（例: `7_data-update_guide.md`）。

- 新規ドキュメントには、`docs/` の既存連番の次の番号を採番します。番号は内容のおおまかな読み順を表すので、関連の強い既存ドキュメントの近くに位置づけられる番号が望ましいです。
- `説明` 部分は内容が推測できる名前にします。既存ファイルのスタイル（`folder-structure`、`setup_guide`、`test-specification` のようなケバブ／スネーク）に寄せ、突飛な命名は避けます。
- 連番に乗らない補助資料（例: `gcp-costs.md`）は既存にも存在します。連番にするか否かは、「開発者が順に読むべき基幹ドキュメントか、随時参照する補足か」で判断します。

### 3. ドキュメントの構造

新規ドキュメントは、冒頭で **何のための文書か（目的）と誰のための文書か（対象読者）** が分かるように始めます。このリポジトリの既存ドキュメントは、見出しの直後に一文の概要を置く形で統一されています。これに合わせてください。

```markdown
# パフォーマンス最適化

このドキュメントは、実装されたパフォーマンス最適化について説明します。

## ...
```

- 1行概要は「このドキュメントは、〜を説明します。」「本ドキュメントでは、〜を記載します。」のような既存の言い回しに揃えます。
- 対象読者が自明でない場合や前提知識が要る場合は、概要に続けて前提・対象を補足します。
- 本文の見出し階層は、同種の既存ドキュメントの粒度に合わせます。

### 4. 索引（folder-structure）を同期する

新規ドキュメントを作成したら、必ず `docs/1_folder-structure.md` の **ファイル一覧および説明** に追記し、ディレクトリ構造のドキュメントを最新に保ちます。これを怠ると索引と実態がズレ、読み手が文書の存在に気づけなくなります。

同様に、コード側でファイル・ディレクトリの追加／削除／移動が起きたときも、`docs/1_folder-structure.md` のツリーと説明を更新します。

### 5. フォーマット

- **Markdown** で記述します。
- 図解が有効な場面（システム構成、データフロー、状態遷移、シーケンスなど）では **Mermaid 記法** を使います。既存の `docs/3_architecture.md` が Mermaid を多用しているので、図のスタイル（`graph TB`、`subgraph`、`style` での着色など）を参考に揃えます。

## コード変更に伴うドキュメント更新

コードの変更を扱ったとき（または変更を把握しているとき）は、関連ドキュメントが古くなっていないかを確認するのも、このスキルの責務です。

- 変更箇所に関係するドキュメント（アーキテクチャ図、セットアップ手順、フォルダ構成、機能説明、データ更新フローなど）を洗い出します。
- 内容がコードの実態とズレていれば、**更新案を提示するか、合意のうえで更新を実行**します。勝手な大改修は避け、何をなぜ直すのかを明示します。
- 新しい機能や概念を導入した場合は、必要に応じて新規ドキュメントを作成し（フロー 2〜4 に従う）、索引も同期します。

## 仕上げ前のセルフチェック

書き終えたら、読み手の視点で次を確認します。

- [ ] 日本語で書かれているか
- [ ] 冒頭に目的（と必要なら対象読者）が書かれているか
- [ ] 既存ドキュメントと用語・トーン・見出し粒度が揃っているか
- [ ] ファイル名が `数字_説明.md` 規則に沿い、連番が適切か
- [ ] 新規作成・ファイル構成変更を `docs/1_folder-structure.md` に反映したか
- [ ] 図が必要な箇所で Mermaid を使い、既存のスタイルに揃っているか
- [ ] コード変更がある場合、影響を受ける既存ドキュメントを確認・更新したか
